## Context

The giscus theme is two committed CSS files under `public/`. They restate the
palette hexes (`--background`, `--foreground`, `--section-swell-*`) that already
live in `src/app/globals.css` (lines 31-60), so they drift on a palette change,
and `globals.css` is the sole source (no TypeScript palette copy exists). The card
`main` background is a flat solid fill.

The repo already runs `tsx` codegen scripts before the build
(`scripts/generate-covers.ts`, `generate-og-images.ts`, `prepare-resume.ts`,
chained in `package.json` `dev`/`build`), and their `public/` outputs are
gitignored and regenerated (e.g. `/public/og/`, `/public/resume-*.pdf`). A giscus
theme generator fits that exact pattern.

## Goals / Non-Goals

**Goals:**
- Single source of truth: the giscus theme derives from `globals.css` tokens.
- Theme-matching gradient card background (section-swell motif).
- Follow the existing generated-`public/`-asset convention (gitignored, built).

**Non-Goals:**
- Refactoring the site palette into a shared TS module (globals.css stays the
  source; the generator parses it).
- Deriving the GitHub prettylights syntax colors from the site palette (those stay
  static GitHub light/dark maps).
- Any change to `useGiscus.ts`, the comment component, or the theme URL scheme.

## Decisions

**1. Parse `globals.css` for the palette rather than introduce a TS palette.**
The user's intent is "generate based on our current colors," and `globals.css` is
already the sole source. The generator reads the file and extracts, scoped to the
`:root {` block (light) and the `:root[data-theme='dark'] {` block (dark), the
`--background`, `--foreground`, and `--section-swell-indigo` tokens, throwing if
any is missing (guards a format change). Alternative considered: a new
`src/config/palette.ts` consumed by both globals.css and the script, rejected
because Tailwind v4 CSS-first cannot import TS, so it would create a second copy,
the opposite of the goal.

**2. Reproduce the current theme in a `buildGiscusTheme(palette, mode)` string
builder.** It emits the same 73 Primer `--color-*` variables and element rules the
committed files have, deriving foreground-tinted `rgba()` values via a
`hex -> {r,g,b}` helper, the inverted primary button (`bg: foreground`,
`text: background`), the `hsl(234 ...)` accent (reusing `pageGradient.ts`
`SATURATION 35 / LIGHTNESS 52` at hue 234, kept as a script constant), and the
static GitHub prettylights maps. Only the `main` background changes (below). Output
is written with `fs.writeFileSync`, mirroring `buildArticleCoverSvg`.

**3. Gradient background from the swell token.** Replace
`main { background: rgba(foreground, 0.025) }` with
`linear-gradient(180deg, <background> 0%, <section-swell-indigo> 55%, <background> 100%)`
plus a `background-color: <background>` fallback, keeping the `foreground/10`
border, `1rem` radius, and padding. This is the same base-to-swell-to-base motif as
`main.home-sections > section` and `.page-gradient` in globals.css, so the card
matches the site.

**4. Gitignore + untrack the two files.** Add them to `.gitignore` and
`git rm --cached` them, so they are build output like `/public/og/`. The deploy
workflow already runs `pnpm build`, which now regenerates them into `./out`.

## Risks / Trade-offs

- **Regex parsing of globals.css is brittle if the file format changes** -> scope
  the match to the known selector blocks and throw on a missing token, so a format
  drift fails the build loudly instead of emitting a wrong theme.
- **Untracking committed files can surprise a checkout that expects them** -> they
  are regenerated by `dev` and `build`; document in the script header. CI builds
  before deploy, so prod always has them.
- **Local visual verification still cannot use the real giscus fetch** (public
  giscus.app iframe is blocked from localhost) -> verify the generated CSS by
  injecting it into the real giscus DOM headless, and confirm the true render on
  the deploy.
- **Generator output must match the current committed CSS (aside from the
  gradient)** -> diff the regenerated files against the previous versions during
  verification; only the `main` background should differ.
